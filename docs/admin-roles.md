# 後台權限角色（編制與開發原則）

> 僅定義**名義編制**與**開發階段方針**；未實作登入／RBAC 前，行為等同擁有者。  
> 讀者：Owner、Agent、日後若要接「Admin 權限矩陣」的客戶。

---

## 角色一覽

| 角色 | 說明 | 權限（規格） |
|------|------|----------------|
| **擁有者（Owner）** | 專案／商家最高權限帳號 | **全部**：POS、後台、主檔 CRUD、庫存寫入、日後報表與設定等，凡系統有的功能皆可使用。 |
| **Admin（管理員）** | 名義上的管理職編制，預留給未來「非擁有者但仍需進後台」的人員 | **尚未訂定**：不寫死任何 allow／deny 清單；客戶提出需求後再開權限矩陣與實作。 |

---

## 開發階段原則（已定案）

1. **一律以擁有者為準**  
   - 開發、測試、Demo、內部使用：視同 **Owner**，不區分 Admin 能做／不能做。  
   - 後台 `/admin`、API 目前無角色判斷時，語意上即 **擁有者權限**。

2. **Admin 僅保留編制名義**  
   - 文件與產品敘述可提及「日後可開 Admin 帳號」，但**不**在現階段實作 Admin 專用限制。  
   - 避免先做半套 RBAC（例如只擋部分 API）卻沒有客戶確認的矩陣。

3. **何時做 Admin 權限**  
   - **待客戶明確提出**「Admin 要能／不能做哪些事」後，再：  
     - 補權限矩陣文件；  
     - 後端 Guard／Policy；  
     - 前端依角色隱藏或停用按鈕。

---

## 與 MVP 身分的關係

- **Phase A（現況）**：無 JWT／無角色欄位 → 實際等同 **單一擁有者情境**（網路隔離當作邊界）。  
- **Phase B（未來）**：若加登入，預設第一個帳號或內建帳號即 **Owner**；**Admin** 帳號可建檔但不套用細粒度權限，直到客戶需求落地。  
  - **可選已實作（後端）**：`ADMIN_API_KEY` + Header 僅區分「有鑰匙／無鑰匙」，**非** RBAC；見 [admin-inventory-ui.md](admin-inventory-ui.md)。
