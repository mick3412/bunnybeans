## 商家電商網站入口參考

本文件為「官網／電商入口」畫面與介面的**開發細節參考**，供前端與後端 Agent 同步構思實作。  
實際 API 合約以 [api-design.md](api-design.md) 為準，型別以 `shared` 為準；業務規格以 [erp-spec.md](erp-spec.md) 為準。

---

## 1. 說明與用途

- 本文件描述「商家電商網站入口」的**頁面結構、區塊、前端元件需求與後端資料/API 需求**，不取代 erp-spec，而是補充官網／電商入口的畫面與介面參考。
- 前端 Agent 可依「區塊 + 前端構思要點」規劃元件、路由與呼叫的 API。
- 後端 Agent 可依「後端構思要點」與 api-design 補充商品／分類／品牌等 draft API，並與 erp-spec 對齊。
- 兩邊以 **api-design.md + shared 型別** 為合約，本文件僅作「入口長什麼樣、需要什麼資料」的參考。

---

## 2. 整體頁面結構（三層）

| 區塊 | 說明 |
|------|------|
| **頂部** | Header：Logo、主導航、會員圖示、購物車圖示（含數量）、搜尋圖示。 |
| **主體** | 依路由切換：首頁內容、或商品列表（左側篩選 + 右側商品網格）、或單一商品頁、會員/購物車等。 |
| **底部** | Footer（選配）；本參考以頂部＋主體為主。 |

---

## 3. 首頁（Landing）參考

### 3.1 區塊列表

- 主 Banner／品牌標語（例如歡迎標題、Slogan）。
- 會員歡迎區：歡迎加入官網專屬會員、註冊禮說明、CTA（例如「加入會員即贈優惠券」）。
- 促銷文案：首購限定、首次下單加贈購物金、會員日紅利雙倍等。
- 側邊或主視覺區：次要標語、JOIN US 等 CTA 按鈕。
- Header 固定：Logo、導航、會員/購物車/搜尋（同 §4）。

### 3.2 前端構思要點

- 需要哪些元件：`Header`、`LandingHero`、`PromoBlock`、`CTAButton` 等。
- 版面配置：主區塊是否需 RWD 斷點、是否需登入狀態判斷（登入後隱藏註冊 CTA 或改為會員專區入口）。
- 靜態內容 vs 動態：哪些區塊從 API 來（精選商品、進行中活動），哪些為靜態或 CMS。

### 3.3 後端構思要點

- 是否需要「首頁專用」API：例如 `GET /promotions/active`、`GET /products/featured`；若無則首頁可先以靜態內容為主。
- 會員註冊禮、首購加贈、紅利規則等，與 erp-spec 的 CRM／行銷活動、會員資產對齊，後續在 api-design 補上對應 API。

## 4. 全站導航與分類參考

### 4.1 主導航項目（示意）

- 新品上架、優惠活動、品牌/服飾等專區、全部商品、商品分類、人氣品牌、其他專區（如貓狗）、專屬會員、實體門市。
- 部分項目為下拉選單：例如「人氣品牌」展開品牌列表，「商品分類」展開階層式分類。

### 4.2 前端構思要點

- 導航資料來源：選單結構、分類樹、品牌列表，可由 API 取得或部分靜態。
- 路由規劃建議：`/`（首頁）、`/products`（全部商品）、`/products/category/:id`、`/products/brand/:id` 或 `/brands/:id`、`/member`、`/stores` 等。
- 下拉選單的展開/收合、階層式分類的展開方式（單層或多層）。

### 4.3 後端構思要點

- `GET /categories`：樹狀結構（id, name, parentId, children），與 erp-spec 商品分類對齊。
- `GET /brands` 或於商品 API 回傳 facet：品牌列表供導航與篩選。
- 新 API 需先在 [api-design.md](api-design.md) 定義為 draft，本文件僅標註「預期會用到的 API 類型與參數方向」。

---

## 5. 商品列表頁參考

### 5.1 版面

- **左側**：階層式商品分類、篩選（如品牌、價格區間）；可與 §4 分類樹一致。
- **右側**：麵包屑或篩選標籤、商品總數（如「共 N 件商品」）、排序下拉（預設排序、價格、上新等）、商品卡網格、分頁（如 1/16）。

### 5.2 前端構思要點

- 網址與 query 對應：`categoryId`、`brandId`、`sort`、`page`、`limit`，與後端 GET /products 參數一致。
- ProductCard 元件欄位：圖片、名稱、價格、標籤（熱賣、獨家、公益等）。
- 篩選狀態管理：左側勾選與 URL 同步，重新整理後可還原狀態。

### 5.3 後端構思要點

- `GET /products`：query 參數建議 `category`、`brand`、`sort`、`page`、`limit`；回傳結構 `{ items: [], total: number }`，與 api-design 未來商品 API 對齊。
- 商品卡所需欄位：id, name, imageUrl(s), price, tags/labels；必要時可加庫存狀態（可售/缺貨）。

---

## 6. 商品卡與單品頁（簡要）

### 6.1 商品卡（列表用）

- 圖片、名稱、價格、標籤（熱賣、獨家、公益等）。
- 點擊進入單品頁（例如 `/products/:id`）。

### 6.2 單品頁

- 詳情、規格、加購選項、加入購物車按鈕。
- 後端需 `GET /products/:id` 與庫存／價格邏輯，與 erp-spec 商品主檔、庫存顯示對齊。

---

## 7. 會員與購物車入口

### 7.1 Header 上的入口

- 會員圖示：未登入導向登入/註冊，已登入導向會員中心。
- 購物車圖示：顯示數量 badge，點擊進入購物車頁。

### 7.2 前端構思要點

- 登入狀態：由 Auth API 或 token 判斷，決定顯示「登入」或「會員」。
- 購物車數量來源：呼叫購物車 API 或與 POS 整合時由同一來源取得。

### 7.3 後端構思要點

- 登入／會員 API：與 erp-spec 消費者／會員管理對齊；具體合約在 api-design 定義。
- 購物車 API：與 erp-spec 前端消費流程、結帳對齊；若官網與門市共用庫存與訂單，需在 api-design 中標註與 POS 的邊界。

## 8. 與現有文件的對應

| 本文件區塊 | erp-spec 對應 | api-design 對應 |
|------------|----------------|-----------------|
| 首頁、促銷 | 消費者/會員、CRM 進階與行銷管理 | 未來活動/優惠 API（draft） |
| 導航、分類、品牌 | 商品與庫存主檔（分類、品牌） | GET /categories、GET /brands、GET /products（draft） |
| 商品列表、單品頁 | 商品與庫存、價格與成本 | GET /products、GET /products/:id（draft） |
| 會員、購物車 | 消費者/會員、前端消費流程 | 會員 API、購物車/結帳 API（draft） |

- 新 API 一律先在 [api-design.md](api-design.md) 定義，本文件僅標註「預期會用到的 API 類型與參數方向」，不重複定義完整合約。
