# 前端全局優化審查（詳細報告）

以下為逐面向的詳細審查結果，每項附檔案路徑。

---

## 一、Code Splitting / Lazy Loading：全部 eager import

### 現狀

[App.tsx](frontend/src/App.tsx) **所有 58 個頁面**均為 static import，0 個 `React.lazy`、0 個 `Suspense`。首次加載 JS bundle 包含全部頁面程式碼。

### 建議

- Admin 區域（約 45+ 頁面）使用 `React.lazy` + `Suspense` 做路由級 code splitting。
- POS 收銀端因效能敏感可保持 eager。
- 至少拆成 3 個 chunk：POS、Admin 主頁、Admin 子功能。

---

仍有大量頁面使用 Tailwind 原色（red-、green-、blue-、amber-、emerald- 等），未用 design token（brand-danger、brand-success 等）：

**重災區**（10+ 處）：

- [AdminReceivingNotesPage.tsx](frontend/src/pages/admin/AdminReceivingNotesPage.tsx) — 38 處
- [AdminCrmJobsPage.tsx](frontend/src/pages/admin/AdminCrmJobsPage.tsx) — 18 處
- [PosOrderDetailPage.tsx](frontend/src/pages/PosOrderDetailPage.tsx) — 19 處
- [AdminPurchaseOrdersPage.tsx](frontend/src/pages/admin/AdminPurchaseOrdersPage.tsx) — 16 處
- [PosPage.tsx](frontend/src/pages/PosPage.tsx) — 14 處
- [AdminInventoryPage.tsx](frontend/src/pages/admin/AdminInventoryPage.tsx) — 11 處
- [AdminCustomersPage.tsx](frontend/src/pages/admin/AdminCustomersPage.tsx) — 8 處

**中度**（3-9 處）：AdminReplenishmentPage、LoyaltyPointLedgerPage、LoyaltyDashboardPage、PosReportsPage、PosCheckoutModal、AdminDispatchRulesPage、AdminProductsPage、AdminOpsReportClicksPage、LoyaltyTierRulesPage、AdminSuppliersPage 等。

### 2-B. neutral- 殘留

- [AdminCustomersPage.tsx](frontend/src/pages/admin/AdminCustomersPage.tsx) L497, 569 — `border-neutral-300`
- [AdminPurchaseOrdersPage.tsx](frontend/src/pages/admin/AdminPurchaseOrdersPage.tsx) L497, 617 — `text-neutral-600`、`border-brand-surface`
- [AdminLayout.tsx](frontend/src/pages/admin/AdminLayout.tsx) L16, 236, 247, 259, 271, 283, 295 — `text-neutral-400`、`text-neutral-500`
- [PosLayout.tsx](frontend/src/pages/PosLayout.tsx) L10, 45 — `text-neutral-400`

### 2-C. slate- 殘留

- [AdminReplenishmentPage.tsx](frontend/src/pages/admin/AdminReplenishmentPage.tsx) L392 — `border-slate-100`

### 2-D. ErrorBoundary 硬編碼

- [ErrorBoundary.tsx](frontend/src/shared/components/ErrorBoundary.tsx) L35-43 — `red-200`、`red-50`、`red-800`、`red-700`、`red-600`、`red-700` 應改用 `brand-danger` token。

### 建議

依 [styles.css](frontend/src/styles.css) 的 design token 映射：

- `red-`* -> `brand-danger`
- `green-`* / `emerald-`* -> `brand-success`
- `amber-`* / `orange-*` -> `brand-warning`
- `blue-*` / `sky-*` -> `brand-primary`
- `slate-*` / `neutral-*` / `gray-*` -> `brand-surface`（邊框）、`content`（文字）、`muted`（次要文字）

## 三、Modal / Drawer 無共用元件

### 現狀

至少 6+ 處手動實作 modal overlay（`fixed inset-0 z-50 flex items-center justify-center bg-black/40`），各自處理 ESC 關閉、點擊外部關閉、aria 屬性：

- [PosCheckoutModal.tsx](frontend/src/pages/PosCheckoutModal.tsx) — 有 role="dialog" aria-modal
- [AdminSuppliersPage.tsx](frontend/src/pages/admin/AdminSuppliersPage.tsx) L272 — 有 role="dialog"，無 aria-modal
- [AdminOpsJobsPage.tsx](frontend/src/pages/admin/AdminOpsJobsPage.tsx) L319 — 有 role="dialog" aria-modal
- [AdminReceivingNotesPage.tsx](frontend/src/pages/admin/AdminReceivingNotesPage.tsx) L636 — 有
- [PosOrderDetailPage.tsx](frontend/src/pages/PosOrderDetailPage.tsx) L275 — 有
- [AdminPurchaseOrdersPage.tsx](frontend/src/pages/admin/AdminPurchaseOrdersPage.tsx) L486 — 有
- [AdminCustomersPage.tsx](frontend/src/pages/admin/AdminCustomersPage.tsx) L490 — **無 role="dialog" 也無 aria-modal**

### 建議

抽出共用 `Modal` 元件到 `shared/components/Modal.tsx`，統一處理：

- overlay backdrop
- ESC 鍵關閉
- `role="dialog"` + `aria-modal="true"` + `aria-labelledby`
- focus trap（可選）
- 動畫 transition

## 五、API 層優化

### 5-A. adminApi.ts 過大（2141 行）

[adminApi.ts](frontend/src/modules/admin/adminApi.ts) 單一檔案 2141 行，含所有 admin API 呼叫、interface 定義。

**建議拆分為**：

- `adminApi.ts` — 通用 request 函式、ApiError 型別
- `inventoryApi.ts` — 庫存相關（balances、events、expiring、replenishment、stocktake）
- `financeApi.ts` — 金流相關（events、summary、balances、periods、audit、snapshots）
- `productApi.ts` — 商品/分類/品牌/標籤
- `merchantApi.ts` — 商家/門市/倉庫
- `customerApi.ts` — 客戶/匯入
- `crmApi.ts` — 分群/發券/job
- `opsApi.ts` — ops job、click audit
- `promotionApi.ts` — 促銷規則
（`purchaseApi.ts` 和 `loyaltyApi.ts` 已獨立，良好）

### 5-B. needsAdminKey 硬編碼路由表

`needsAdminKey()` 函式（L11-76）以硬編碼 if-else 判斷哪些路徑需要 admin key，維護困難。

**建議**：改為統一規則——所有非 GET 請求都帶 admin key，或使用 allowlist pattern。

### 5-C. 缺少 request deduplication / 防重複提交

多處 create/submit 操作無 debounce 或 disable-on-submit 保護。搜尋僅 AdminPurchaseOrdersPage 有提到 debounce（L130 comment），但實際僅有 1 處。

### 5-D. useDefaultMerchantId 每次 mount 重複請求

[useDefaultMerchantId.ts](frontend/src/shared/hooks/useDefaultMerchantId.ts) 每次使用此 hook 的元件 mount 時都會重新發請求。應改為全域快取（Context 或 module-level cache）。

## 六、效能優化

### 6-A. 搜尋缺少 debounce

幾乎所有搜尋/篩選的 `onChange` 直接觸發 API 呼叫，無 debounce。尤其：

- AdminProductsPage 搜尋
- AdminCustomersPage 篩選
- AdminSuppliersPage 搜尋
- PosPage 商品搜尋

### 6-B. 大列表缺少虛擬化

商品列表、客戶列表等無分頁限制時可能回傳大量資料，UI 未使用 virtualization（如 react-window）。

### 6-C. useMemo / useCallback 使用量

已有相當程度使用（45 個頁面中多數有 useMemo/useCallback），狀況尚可。需關注 AdminDashboardPage（8 個 useEffect）是否有多餘 re-fetch。

### 6-D. formatMoney / formatInt 重複定義

`formatMoney` 在多個頁面內重複定義（AdminDashboardPage、PosReportsPage、PosOrderDetailPage、AdminReportsPage 等至少 6+ 處各自定義了 `Intl.NumberFormat` 格式化）。

**建議**：抽出到 `shared/utils/formatMoney.ts`。

## 七、Accessibility

### 7-A. Modal 缺少統一 a11y

如 §三 所述，部分 modal 缺少 `aria-modal`、`aria-labelledby`、ESC 鍵監聽、focus trap。

### 7-B. focus ring 覆蓋率

多數表單元素已有 focus-visible:ring（良好）。shared/Button 和 TextInput 已完善。但部分頁面的原生 `<select>`、`<input>` 缺少：

- AdminCustomersPage L497 — `<select>` 未加 focus ring class

### 7-C. 缺少 aria-label

搜尋 icon input 組合普遍缺少 `aria-label` 或 `<label>` 關聯。

### 7-D. @ts-expect-error

僅 1 處：[AdminPromotionEditPage.tsx](frontend/src/pages/admin/AdminPromotionEditPage.tsx) L207 — 可定義型別消除。

## 八、錯誤/空態/載入 一致性

### 已完善

- `EmptyState` 元件已在 13 個頁面使用。
- `Alert variant="error"` 已在 26 個頁面使用。
- `StandardListLayout` 內建 loading/error/empty 三態。

### 仍需改善

- 頁面未使用 StandardListLayout 的 loading 顯示不一致：有些是 "載入中…"、有些是骨架（較少）、有些完全無。
- 部分大型頁面（AdminReceivingNotesPage、AdminPurchaseOrdersPage）的 error 回饋分散在多處，未統一走 Alert。
