# 後端全局優化審查（詳細報告）

以下為逐模組、逐面向的詳細審查結果，每項均附檔案路徑與行號。

---

## 一、Guard 一致性：缺少 AdminApiKeyGuard 的 Controller

以下 Controller 的**所有**寫入端點均未加 `@UseGuards(AdminApiKeyGuard)`，屬於安全漏洞：

- **MerchantController**（[merchant.controller.ts](backend/src/modules/merchant/interface/merchant.controller.ts)）— 所有 CRUD（merchants、stores、warehouses）0 個 Guard
- **PurchaseOrderController**（[purchase-order.controller.ts](backend/src/modules/purchase/interface/purchase-order.controller.ts)）— 所有端點 0 個 Guard
- **ReceivingNoteController**（[receiving-note.controller.ts](backend/src/modules/purchase/interface/receiving-note.controller.ts)）— 所有端點 0 個 Guard
- **SupplierController**（[supplier.controller.ts](backend/src/modules/purchase/interface/supplier.controller.ts)）— 所有端點 0 個 Guard
- **PurchaseReportsController**（[purchase-reports.controller.ts](backend/src/modules/purchase/interface/purchase-reports.controller.ts)）— 0 個 Guard
- **DashboardController**（[dashboard.controller.ts](backend/src/modules/dashboard/interface/dashboard.controller.ts)）— 0 個 Guard
- **PosController** 部分端點 — `POST /`, `POST :id/payments`, `POST :id/refunds`, `POST :id/return-to-stock`, `GET /`, `GET :id` 均無 Guard（僅 `GET /export` 有）

> 建議：至少所有寫入端點（POST/PATCH/DELETE）加 Guard；或改為 class-level `@UseGuards(AdminApiKeyGuard)`。

## 二、錯誤處理：throw-exceptions 工廠遷移

`throw-exceptions.ts` 定義了 `throwBadRequest`、`throwNotFound`、`throwConflict`，但目前**僅 2 個 controller 使用**：

- [promotion.controller.ts](backend/src/modules/promotion/interface/promotion.controller.ts) L32, L96
- [pos-reports.controller.ts](backend/src/modules/pos/interface/pos-reports.controller.ts) L21

以下模組仍使用 `new BadRequestException(...)` / `new NotFoundException(...)` 直接 throw，需遷移：

| inventory   | inventory.service.ts                                                      | ~30+ 處                                                      |
| finance     | finance.service.ts                                                        | ~15+ 處                                                      |
| purchase    | purchase-order.service.ts, receiving-note.service.ts, supplier.service.ts | ~40+ 處                                                      |
| product-tag | product-tag.service.ts                                                    | ~10+ 處                                                      |
| category    | category.service.ts                                                       | ~10+ 處                                                      |
| brand       | brand.service.ts                                                          | ~10+ 處                                                      |
| loyalty     | loyalty.service.ts                                                        | ~5+ 處                                                       |

## 三、DTO / ValidationPipe 擴展

目前全域 `ValidationPipe` 已啟用（[main.ts](backend/src/main.ts) L41-47），但**僅 3 個 DTO** 存在：

- `backend/src/modules/promotion/dto/create-promotion-rule.dto.ts`
- `backend/src/modules/promotion/dto/update-promotion-rule.dto.ts`
- `backend/src/modules/pos/dto/pos-reports-query.dto.ts`

其餘所有 Controller 的 `@Body()` 參數仍為 inline interface 或裸型別，手動驗證散佈在 service 層。以下為高優先 DTO 缺口：

- **PosController.create** — body 有 10+ 欄位，驗證全在 pos.service.ts
- **PurchaseOrderController.create / quickReceive** — body 有 nested lines 陣列
- **ReceivingNoteController.create / patchLines** — body 有 nested lines
- **CustomerController.merge / create / update** — body 有多個必填欄位
- **InventoryController** — body 參數散落
- **MerchantController** — 完全無驗證（直接 pass-through 給 service）
- **FinanceController** — L68 手動 try/catch parseInt（應改 DTO `@Transform`）

## 四、查詢效能

### 4-A. N+1 與串行可並行化

1. `**pos.service.ts` createOrder — 多倉庫存檢查**（L194-214）
  - 巢狀 `for (wh) { for (item) { await findUnique } }` — 每個 item 一次 DB 查詢
  - 優化：改為單次 `findMany({ where: { productId: { in: ... }, warehouseId: wh.id } })`
2. `**pos.service.ts` createOrder — 庫存扣減**（L299-308）
  - `for (item) { await recordInventoryEvent }` — 串行逐筆寫入
  - 優化：可並行 `Promise.all` 或合併為批次寫入
3. `**pos.service.ts` createOrder — 信用付款記帳**（L326-337）
  - `for (p) { await recordFinanceEvent }` — 串行
  - 優化：`Promise.all`
4. `**pos.service.ts` returnToStock — 逐筆退庫**（L579-621）
  - `for (row) { await recordInventoryEvent }` — 串行
  - 優化：`Promise.all`
5. `**customer.service.ts` getConsumptionInsights**（L99-181）
  - 4 次串行查詢（lastOrder, firstOrder, aggregate, count）+ 1 次 raw SQL
  - 優化：前 4 次可 `Promise.all`
6. `**customer.service.ts` getById**（L345-397）
  - 5 次串行查詢（findUnique, pointLedger findFirst, loyaltySettings findUnique, pointLedger findFirst, getConsumptionInsights）
  - 優化：pointLedger 和 loyaltySettings 可並行
7. `**customer.service.ts` listByMerchant**（L217-341）
  - findMany 取所有客戶 -> $queryRaw 取積分 -> loyaltySettings findUnique -> pointLedger groupBy
  - 可部分並行（settings + balances 同時查）
8. `**pos.service.ts` phoneDigitsComparable 模糊查**（L138-162）
  - `findMany({ phone: { not: null } })` 取全部再 JS filter — 全表掃描
  - 優化：改為 SQL LIKE 或正規化欄位

### 4-B. 缺少 Transaction 的多步驟寫入

1. `**pos.service.ts` createOrder**（L286-363）
  - 建單 + 扣庫存 + 記帳 + 忠誠度 — 全部串行無 Transaction
  - 若中途失敗，庫存已扣但帳未記，資料不一致
  - **高風險**：應包在 `$transaction` 中
2. `**pos.service.ts` refundToOrder**（L444-515）
  - 查單 + 查已退金額 + 記帳 — 無 Transaction
  - 並行退款可能超額
3. `**pos.service.ts` appendPaymentToOrder**（L366-441）
  - 查單 + 查應收 + 建付款 + 記帳 — 無 Transaction
4. `**pos.service.ts` returnToStock**（L518-625）
  - 查單 + 逐筆退庫 — 無 Transaction

### 4-C. 未使用 select 的 findMany

- `product.repository.ts` `findAll`（L59）— `findMany` 無 `select`，回傳所有欄位（大列表場景浪費）
- `pos.repository.ts` `findManyForExport`（L139）— `include: { customer: true }` 含完整 customer 欄位
- `pos.service.ts` L138 — `customer.findMany({ phone: { not: null } })` 取全表全欄位
- `customer.service.ts` L247 — findMany 已有 select（OK）

## 七、缺少分頁的列表 API

以下列表端點回傳全量而無分頁，大數據量時有效能風險：

- `GET /customers` — `customer.service.ts listByMerchant` 無分頁
- `GET /products` — `product.repository.ts findAll` 無 take/skip
- `GET /merchants`、`GET /stores`、`GET /warehouses` — 無分頁（量小暫可忽略）
- `GET /suppliers` — 確認是否無分頁

## 八、測試覆蓋缺口

現有 15 個 integration-spec 檔案。以下模組**完全無測試**：

- **merchant** — 無 integration-spec
- **dashboard** — 無 integration-spec

以下為缺少的 edge case：

- `pos.service.ts` createOrder — 並行建單的庫存競爭（無 Transaction）
- `pos.service.ts` refundToOrder — 並行退款超額
- `customer.service.ts` merge — 合併後積分/訂單歸屬

## 九、logging 改善

`console.warn` 僅用在 integration-spec 跳過提示（OK）。production code 已全面使用 NestJS `Logger`（[main.ts](backend/src/main.ts) L9, [http-exception.filter.ts](backend/src/shared/filters/http-exception.filter.ts) L23）。

改善空間：

- 重要業務操作（createOrder, refund, returnToStock, merge customers）可加 Logger.log 記錄操作摘要（traceId + 關鍵參數）

## 十、其他優化

1. **product.repository.ts findAll 無 take 上限** — 大量 Product 時回傳過大
2. **pos.repository.ts findLineRowsForExport** — while 迴圈 + 逐批查詢可改為 cursor-based pagination
3. **customer.service.ts L138-162 全表掃描手機查詢** — 應改為 DB-side 正規化或 SQL LIKE
4. **promotion.controller.ts L66-67** — `conditions` 與 `actions` 以 `as any` 傳入，應定義為 DTO class 搭配 `@ValidateNested`
5. **dashboard.service.ts** — 5 次 DB 查詢（已用 Promise.all，OK）；但可考慮加記憶體快取（TTL 1-5 分鐘）
